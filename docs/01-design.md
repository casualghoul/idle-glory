# Design: Glory — a WW1 idle war empire for your terminal

## Problem Statement

Build a terminal-native idle/incremental game (Cookie Clicker / AdVenture
Capitalist lineage — numbers go up, buy upgrades to make numbers go up faster)
that is *fun* and that hones the builder's game-dev skills plus their long-standing
itch to learn a lower-level/systems-ish language. The game is designed to live in
the same terminal developers already stare at while AI coding agents (Claude Code,
etc.) run — played in the dead time during agent runs.

Theme: WW1-era war empire. Grow an army, upgrade weapons, build an empire, crush
enemy nations. "Glory to the empire." No world map needed. Simple ASCII graphics,
but *satisfying animations* are the explicit differentiator.

## What Makes This Cool

The core delight is **a war that thumps in your terminal** — pressing a key to
launch an offensive and watching shells arc across no-man's-land, impacts flash,
the enemy line recede, numbers pop. ASCII, but with weight and motion.

The deeper novelty is the **native habitat / "reward the glance" inversion** (the
EUREKA from this session):

> Conventional idle games treat themselves as the *primary* activity and fight for
> your attention with notifications, FOMO timers, and dark patterns. Glory is
> explicitly the *secondary* activity — it exists to fill dead time you already
> have, in a terminal you're already in. That inverts the design. You don't need
> manipulative re-engagement; the player is structurally already present. The goal
> isn't "pull them back," it's "reward the glance."

The flagship expression of this (post-core) is **terminal-activity sync**: the war
runs "hot" while your AI agent is working and an enemy push hits when the agent goes
idle — i.e. the game wants your eyes at the exact moment your AI needs them again.

## Constraints

- **Language:** Go. Chosen over Rust/C/C++ for fastest time-to-fun while still
  teaching real systems-ish skills (goroutines/channels, Elm-style state). It's a
  learning project; momentum keeps it alive.
- **Run model:** Live TUI in a terminal pane (tmux split / second tab). Real-time
  animated, watched while AI works. Not a CLI-poke and not launch-on-demand.
- **Graphics:** ASCII only. Simplicity is fine; *feel* is not negotiable.
- **Scope philosophy:** Boil the ocean toward the full layered idle game, but ship
  in independently-playable milestones.
- **Single-developer side project.** No deadline, no team. Fun is the success metric.

## Premises

All six confirmed by the builder this session:

1. **"Reward the glance" is the prime directive.** Every screen delivers something
   satisfying in a 5-second look. No nagging, no dark patterns.
2. **State persists; away-progress is computed.** Saves to disk (e.g.
   `~/.config/glory/save.json`); on relaunch computes "while you were away,
   your factories produced X." Non-negotiable for an idle game.
3. **Vertical slice first, then layer.** Ship a satisfying core (passive
   accumulation + upgrade tree + one animated offensive) before prestige eras and
   attrition. Each layer is playable on its own.
4. **The economy is data, not hardcoded.** Cost/reward curves live in tunable
   config so balancing is iteration, not a rewrite. (Caveat below: NOT in the
   weekend spike — hardcode there.)
5. **Animation/feel is a first-class system, built early** — it IS the
   differentiator, so it gets real design attention up front.
6. **Distribution = single Go binary.** `go install` + GitHub Releases with
   prebuilt Linux/macOS binaries. One command to install and run `glory`.

## Cross-Model Perspective

An independent Claude subagent (no memory of the conversation, given only a
structured brief) gave a cold read. Key points, close-paraphrased:

- **Coolest unconsidered version:** make terminal-activity the war's *tempo* —
  hot when the agent runs, an enemy push when it goes idle. The downtime framing
  becomes a literal mechanic, not just a tagline. Ship behind an optional toggle so
  it degrades to a plain idle game.
- **The one make-or-break thing:** "animation/feel is a first-class system" — a
  WW1 idle game with tunable curves is a commodity genre; the *only* stated
  differentiator is satisfying ASCII animation. So make-or-break = does a single
  offensive *resolving on screen* feel good enough to want to trigger one and watch.
  **Build the battle animation before the economy.**
- **The 50% project:** the Charm stack — Bubble Tea (Elm loop + `tick` frame
  clock), Lip Gloss (styled ASCII layout), and especially **Harmonica** (Charm's
  spring-physics lib — eased, springy motion almost nobody uses for games). The 50%
  still to build: deterministic away-progress math, save/load + schema migration,
  the economy sim, and the *choreography* of a battle (sequencing, impact,
  screen-shake via column offset, ASCII particles).
- **One-weekend slice:** a single front, a single `[F] Fire Offensive` key, one
  great battle animation. Top = live-ticking munitions line; middle = ASCII
  no-man's-land (your trench left, enemy right); bottom = fire key + charge meter.
  Press F → choreographed barrage, enemy line recedes, screen-shake, numbers pop,
  2-3s, make it THUMP. Cut everything else; hardcode three constants. Caution:
  don't build a config/economy loader this weekend.

**Synthesis (where the lead agent agreed/pushed back):**
- Agreed and *upgraded* premise #5 to a validation gate: the battle animation is
  the first thing built, before the economy.
- Adopted Harmonica as a named dependency — cheapest path from "ASCII moves" to
  "ASCII feels good."
- Kept activity-sync as a **flagship post-core feature behind a toggle**, not a
  week-1 requirement.
- Push-back: the subagent's "cut save/load + offline progress" applies to the
  *throwaway spike* only. The real v1 vertical slice adds persistence + away-progress
  back (premise #2). Two distinct milestones — see below.

## Approaches Considered

### Approach A: Thump-First — Spike, then Vertical Slice, then Layer (CHOSEN)
Weekend 0 is a throwaway, hardcoded animation spike to prove one offensive *feels*
good. Only if it grins do you build the real v1 (accumulation + upgrade tree +
animated battles + persistence + away-progress), then layer prestige eras &
attrition. Effort S to validate / M for v1 / L for full vision. Risk Low — de-risks
the make-or-break thing first. Cons: spike code is partly throwaway (intentional).

### Approach B: Engine-First — Headless Sim Core, then Render Layer
Build a clean pure-Go game core first (deterministic tick sim, economy math,
save/load, away-progress, all unit-testable with zero TUI), then build Bubble Tea
views on top. Effort M before anything visible / L full. Risk Med — builds
infrastructure before validating fun. Best long-term architecture and "real
engineering" learning, but delays the grin and risks over-engineering an unproven game.

### Approach C: Activity-Sync Showcase — Lead with the Signature Mechanic
Build the terminal-activity-sync centerpiece early; organize the whole game around
mirroring the agent's tempo. Effort L. Risk High — depends on a clean cheap activity
signal and front-loads the gimmick before the core loop is proven fun. Most unique
and shareable, but riskiest.

## Recommended Approach

**Approach A (Thump-First).** The project's success hinges on one question: does
watching an offensive resolve in ASCII make you grin? A answers that in a weekend,
before a month is spent on economy curves and prestige. It also best fits the goal —
fast, felt progress keeps a side project alive.

**Grafted-in discipline from B:** even in the slice, keep battle resolution and
economy as *pure, testable Go functions* (`ResolveBattle(state) Outcome`,
`Tick(state, dt) state`) separated from the Bubble Tea view. Cheap to do, huge
payoff for tuning and for the systems-engineering learning goal.

**C is deferred, not dropped:** activity-sync is the flagship feature added once the
core loop is proven fun, behind a `--sync-terminal` toggle.

## Open Questions

1. **Activity signal source** (for the later sync feature): sentinel file the agent
   touches vs. CPU/proc polling vs. watching a log file (fsnotify). Decide when you
   reach that milestone; sentinel file is likely the cleanest and most portable.
2. **Save format & migration:** JSON is the obvious start. Add a `version` field
   from day one so schema migration is painless when you add prestige/attrition.
3. **Prestige curve shape:** what exactly carries over across eras (permanent
   multipliers? unlocked unit types? "war bonds"?). A balancing question for the
   layering phase, not the slice.
4. **Color policy:** Lip Gloss supports 256-color/truecolor, but many dev terminals
   vary. Decide a tasteful 16-color-safe palette with optional truecolor enhancement.

## Success Criteria

- **Spike (Weekend 0):** pressing `F` to fire an offensive and watching it resolve
  makes *you* grin. Binary yes/no. That's the whole gate.
- **v1 Vertical Slice:** one front, live-ticking resources, an upgrade tree that
  visibly speeds things up, animated offensives, save/load, and an away-progress
  summary on relaunch. Playable for 15+ minutes without boredom.
- **Full vision:** prestige eras (1914 → 1916 → escalating) with permanent bonuses,
  light attrition pressure, multiple enemy nations, and the optional terminal-sync
  flagship feature. Installable via one command.

## Distribution Plan

- **Artifact:** a single statically-linked Go binary named `glory`.
- **Install paths:** `go install github.com/<you>/glory@latest` for Go users;
  prebuilt binaries (linux-amd64, linux-arm64, darwin-arm64, darwin-amd64) attached
  to **GitHub Releases**.
- **CI/CD:** GitHub Actions workflow on tag push — build the matrix with
  [GoReleaser](https://goreleaser.com/) (handles cross-compile, archives, checksums,
  release notes, and a Homebrew tap if you want `brew install glory` later).
- **Config/saves:** XDG paths — `~/.config/glory/` for config, `~/.local/state/glory/`
  (or `~/.config/glory/save.json`) for saves.

## Next Steps

**This weekend (the spike — throwaway, hardcoded, one job: prove the thump):**
1. `go mod init`, add Bubble Tea + Lip Gloss + Harmonica.
2. Static layout (Lip Gloss panes): top munitions line ticking via a `tick` cmd;
   middle no-man's-land (your trench `|‾‾|`, enemy trench, dotted gap); bottom
   `[F] Fire Offensive` + a charge meter filling from munitions.
3. The only real work: the `F` choreography. Shells arc across the gap — note
   Harmonica is a *spring* solver, not a *projectile* solver: drive x with a
   spring/lerp toward the target and y with a separate parabola (or chained
   spring targets) to actually get the arc; a lone spring on (x,y) gives a
   straight diagonal, not a ballistic curve, and the arc is the money shot.
   Then impacts flash, enemy line takes damage and recedes, screen-shake (offset
   the *entire rendered view* by ±N columns for a few frames), numbers pop.
   2-3 seconds. Iterate on feel until *you grin*.
4. Resolution = hardcoded `yourPower vs theirPower` → push line forward or get
   repelled. Three hardcoded constants. No config, no save, no offline progress.

**Then (v1 vertical slice — only if the spike grins):**
5. Extract `Tick(state, dt)` and `ResolveBattle(state)` as pure testable functions
   (graft from Approach B). Add unit tests.
6. Upgrade tree (a handful of WW1 upgrades: artillery, rifles, supply lines, a tech
   that ups passive munitions/s). Spending visibly changes the tick rate.
7. Save/load to `~/.config/glory/save.json` with a `version` field.
8. Away-progress: on launch, compute earnings since last save, show a "while you
   were away..." summary. **Guard the time delta** — clamp `now - lastSave` to
   `[0, cap]` so a backward system clock can't produce negative/garbage earnings
   and a 3-week-asleep laptop can't overflow into an instant win. Classic
   idle-game footgun; handle it here, not later.

**Later (layer the full vision):**
9. Prestige eras ("sue for peace" → reset into a harder war with permanent bonuses).
10. Light attrition pressure so idling has a cost.
11. Multiple enemy nations / fronts.
12. Flagship: optional terminal-activity sync (`--sync-terminal`) — war runs hot
    when your agent works, enemy pushes when it idles.
13. Distribution: GoReleaser + GitHub Actions, tag your first release.

## What I noticed about how you think

- You didn't say "a war game." You said "older war vibes, like World War 1... grow
  an army, upgrade weapons, build an empire, crush your enemies. glory to the
  empire." That specificity — a *period*, a *fantasy*, even a battle cry — is the
  difference between a theme and a reskin. You're designing the feeling, not the
  feature list.
- You corrected my assumption mid-question: "i don't think we necessarily need
  maps." Cutting your own scope before a single line of code is written is a rare
  instinct. Most people add; you subtracted toward the thing that matters.
- "the graphics can be simple ASCII, but i want the animations to be satisfying."
  You separated *fidelity* from *feel* — knowing those are different axes is exactly
  the taste that makes the make-or-break call here. The independent reviewer
  landed on the same point with no prompting.
- You picked Go over Rust/C "to get to fun fast" while still scratching the
  lower-level itch. That's a builder optimizing for momentum over purity — the right
  call for a side project that needs to survive contact with your weekends.
